# Alexander Montilla — Portfolio

Personal portfolio for Alexander Montilla, a Full Stack Developer based in the
Dominican Republic. The site presents selected projects, professional
experience, education, technical skills, inspiration, and curated development
resources.

[View the live portfolio](https://alexandermontillarivera.com)

## Overview

This project is built as a server-rendered Astro application with React
components where client-side interactivity is useful. Its visual system supports
light and dark themes, responsive layouts, accessible navigation, and
motion-reduction preferences.

The portfolio also includes an optional Spotify integration. When music or a
podcast is actively playing, a floating mini-player appears at the bottom of the
page. It can be minimized by the visitor and remains completely hidden when
nothing is playing or Spotify is unavailable.

## Highlights

- Responsive portfolio layout for desktop, tablet, and mobile.
- Light and dark themes with the selected preference stored locally.
- Gesture-driven mobile navigation presented as a bottom sheet.
- Filterable technical-skills directory.
- Responsive inspiration carousel with normalized imagery.
- Floating Spotify now-playing widget with a persistent minimized state.
- Scroll-progress button for quickly returning to the top of the page.
- Optional WordPress.com blog integration.
- Vercel Analytics and Speed Insights.
- Reduced-motion, reduced-transparency, keyboard, and focus-state support.

## Local Development

Clone the repository and install the dependencies:

```bash
git clone https://github.com/alexandermontillarivera/portfolio-web.git
cd portfolio-web
npm install
```

Create the local environment file:

```bash
cp .env.example .env
```

On Windows PowerShell:

```powershell
Copy-Item .env.example .env
```

Start the development server:

```bash
npm run dev
```

The site will be available at `http://localhost:4321`.

## Environment Variables

All integrations are configured through `.env`. The file is ignored by Git and
must never be committed.

| Variable | Required | Description |
| --- | --- | --- |
| `SPOTIFY_CLIENT_ID` | For Spotify | Client ID from the Spotify Developer Dashboard. |
| `SPOTIFY_CLIENT_SECRET` | For Spotify | Client secret from the Spotify Developer Dashboard. |
| `SPOTIFY_AUTHORIZATION` | For Spotify | OAuth refresh token generated by `npm run spotify:auth`. |
| `SPOTIFY_REFRESH_TOKEN_ISSUED_AT` | Recommended | ISO timestamp written by the authorization helper. |
| `WORDPRESS_SITE_URL` | For the blog | WordPress.com site hostname used by the public REST API. |
| `APP_HOST` | For the blog | Public application origin used for absolute fallback-image URLs. |

Example:

```env
SPOTIFY_CLIENT_ID=""
SPOTIFY_CLIENT_SECRET=""
SPOTIFY_AUTHORIZATION=""
SPOTIFY_REFRESH_TOKEN_ISSUED_AT=""
WORDPRESS_SITE_URL=""
APP_HOST="http://localhost:4321"
```

## Spotify Setup

### 1. Create or configure the application

Create an application in the
[Spotify Developer Dashboard](https://developer.spotify.com/dashboard) and add
this exact redirect URI:

```text
http://127.0.0.1:4321/callback
```

The hostname, port, protocol, and callback path must match exactly. Add the
Spotify account that will authorize the portfolio to the application's user
allowlist when the application is running in Development Mode.

### 2. Add the credentials

Copy the application's client ID and client secret into `.env`:

```env
SPOTIFY_CLIENT_ID="your-client-id"
SPOTIFY_CLIENT_SECRET="your-client-secret"
```

Do not manually create `SPOTIFY_AUTHORIZATION`.

### 3. Generate the refresh token

Stop the Astro development server before running the authorization command,
because the helper temporarily uses port `4321`:

```bash
npm run spotify:auth
```

The helper:

1. Opens Spotify's authorization page.
2. Receives the OAuth callback locally.
3. Exchanges the authorization code for a refresh token.
4. Writes `SPOTIFY_AUTHORIZATION` and
   `SPOTIFY_REFRESH_TOKEN_ISSUED_AT` to `.env`.

The token is not printed to the terminal.

### 4. Verify the integration

```bash
npm run spotify:check
```

This command verifies that the refresh token can obtain an access token and
reports its estimated remaining lifetime. If Spotify revokes or expires the
refresh token, run `npm run spotify:auth` again.

### Runtime behavior

- The browser requests `/api/spotify/now-playing` every 30 seconds.
- Access tokens are generated server-side and cached until shortly before
  expiration.
- Credentials and refresh tokens are never sent to the browser.
- Tracks and podcast episodes are supported.
- A `204`, inactive playback, invalid configuration, or Spotify error produces
  an idle response and keeps the widget hidden.

## WordPress Blog

The repository contains a WordPress.com service and dynamic blog route. The blog
section is currently disabled on the home page, but it can be enabled from
`src/pages/index.astro`.

To use it, provide:

```env
WORDPRESS_SITE_URL="example.wordpress.com"
APP_HOST="http://localhost:4321"
```

The integration uses the public WordPress.com REST API and does not require a
WordPress access token for public posts.

## Production Build

```bash
npm run build
```

Astro uses the Vercel adapter for the production server build. The Node adapter
is selected only by the local preview scripts.

## Deployment

The project is configured for server-side deployment on Vercel:

1. Import the GitHub repository into Vercel.
2. Add the required environment variables in the project settings.
3. Select a supported Node.js runtime; Node.js 22 is recommended for this
   repository.
4. Deploy the production branch.

Spotify credentials must be configured as server-side environment variables in
every Vercel environment where the now-playing widget should work.
